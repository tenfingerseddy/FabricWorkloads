The Problem

Fabric monitoring has critical blind spots

Microsoft Fabric is powerful, but its built-in monitoring leaves platform teams without the visibility they need when it matters most.

30-Day Retention Limit

Monitoring hub and workspace monitoring data disappears after 30 days. Quarter-over-quarter analysis, capacity planning, and audit compliance are impossible without historical data.

No Cross-Item Correlation

A slow pipeline triggers a failed notebook that breaks a semantic model refresh. In Fabric, these are three disconnected events. Root cause analysis takes hours, not minutes.

No SLO Framework

There is no way to define or track service level objectives. When leadership asks about data freshness SLAs or error budgets, you are stuck building manual spreadsheets.

No Proactive Alerting

Fabric has no native alert rules for pipeline failures, SLO breaches, or duration regressions. Your stakeholders find out about data issues before your team does.

Features

Everything Fabric monitoring should have been

Six capabilities that give your platform team complete visibility into health, performance, cost, and incidents.

Long-Retention Event Store

Store monitoring hub events and workspace monitoring data for up to 365 days in a Fabric Eventhouse. Full KQL query access for historical analysis, capacity planning, and compliance auditing.

Cross-Item Correlation

Automatically link related events across pipelines, notebooks, dataflows, and semantic model refreshes. Trace failures from upstream trigger to downstream impact in a single timeline.

SLO Dashboards

Define service level objectives for freshness, success rate, and duration. Track compliance in real time with error budget burn-down, breach prediction, and per-item health status grids.

Proactive Alerting

Define alert rules that catch issues before your stakeholders notice. Detect SLO breaches, duration regressions, consecutive failures, and "likely to breach" conditions with configurable thresholds.

CU Waste Analysis

Per-item compute waste breakdown covering failed retries, duration regressions, and duplicate runs. Monthly projections translate CU consumption into dollar amounts you can reclaim.

Full-Text Event Search

Search across all historical events with filters by item type, status, workspace, and date range. Find that one failed refresh from three months ago in seconds. No 100-activity limit.

How It Works

Up and running in three steps

No agents, no sidecars, no external infrastructure. Everything runs natively inside your Fabric tenant using scheduled notebooks and an Eventhouse.

Step 1

Install

Clone the repo and deploy notebooks to your Fabric workspace. Create an Eventhouse and KQL database for long-retention storage. Takes under five minutes.

Step 2

Configure

Point the ingestion notebook at your workspaces. Define SLO targets and alert thresholds. Scheduled notebooks run automatically every 5 to 15 minutes.

Step 3

Monitor

View health dashboards, SLO compliance, incident timelines, and CU waste from the workload UI. Receive alerts through email, Teams, or your preferred channel.

Built for Fabric

Native integration. Zero data egress. Full permission inheritance.

Unlike external observability tools that require data extraction, Observability Workbench runs entirely inside your Fabric tenant using Fabric-native components.

Zero Data Egress

Your monitoring data never leaves your Fabric tenant. Events are collected, stored, correlated, and queried entirely within Fabric using Eventhouse and KQL. No external API calls, no data pipelines to third-party services.

Permission Inherited

Leverages Entra ID and Fabric's existing RBAC. Users see only the workspaces and items they already have access to. No separate permission model to configure or maintain.

Monitoring Hub Native

All workload jobs appear in Fabric's monitoring hub alongside your existing pipelines, notebooks, and dataflows. One place to see operational status across every Fabric item type.
