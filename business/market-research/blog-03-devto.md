Every Fabric team knows about the visible costs: your capacity SKU, the monthly Azure bill, the burst charges when Spark jobs run hot. These are the costs you plan for.

But there is another category of cost that never appears in any budget: **the CU consumption generated by data quality failures.** Failed pipelines that retry. Notebooks that reprocess entire datasets because upstream data arrived late or corrupted. Semantic model refreshes that run on stale data, triggering manual re-runs when a stakeholder notices at 10 AM.

Based on our analysis of real Fabric workspaces, this hidden cost is significantly larger than most teams estimate.

## How Fabric CU Billing Works (Quick Primer)

Every operation in Fabric -- pipelines, notebooks, Spark queries, semantic model refreshes, Dataflow Gen2 transformations -- consumes **Capacity Units (CUs)** proportional to compute resources and runtime.

Key mechanics:
- **Per-second billing**: A notebook running 300 seconds on 8 CUs = 2,400 CU-seconds
- **Smoothing**: CU consumption is smoothed over time windows (5 min for interactive, 24 hours for background)
- **Throttling**: When consumption exceeds capacity, background jobs are delayed, then rejected
- **Bursting**: Temporary spikes borrow from future capacity, creating "debt"

The critical insight: **wasted CU consumption doesn't just waste money -- it steals capacity from legitimate workloads.** A pipeline that retries three times doesn't just cost 3x; it can push your workspace into throttling, which impacts *every other workload* on the same capacity.

## The Five Categories of CU Waste

### 1. Retry Waste: The Failed Pipeline Loop

A daily pipeline on F64 capacity ($11.52/hour pay-as-you-go):

| Scenario | Monthly CU Cost |
|----------|-----------------|
| Healthy (no failures) | ~$62 |
| 15% failure rate + 1 retry | ~$71 |
| 15% failure rate + 2 retries | ~$81 |
| 15% failure + cascading downstream | ~$120 |

A single pipeline with a 15% failure rate and cascade effects costs **nearly 2x its expected cost.** Multiply across 20 pipelines and you're looking at hundreds of dollars per month in pure waste -- on a single capacity.

### 2. Stale Data Reprocessing: The "Run It Again" Tax

When a pipeline fails silently at 2 AM:

1. Pipeline fails (CU consumed: **wasted**)
2. Downstream notebook runs on stale data (CU consumed: **wasted**)
3. Semantic model refreshes from stale Lakehouse (CU consumed: **wasted**)
4. Stakeholder notices at 10 AM, engineer manually re-runs everything (CU consumed: **duplicated**)

You paid double. And the ad-hoc re-runs compete with scheduled workloads for capacity during business hours.

### 3. Schema Drift Cascades

Dataflow Gen2 has a known behavior: when upstream schemas change, destination tables can be dropped and recreated. Relationships, measures, and calculated columns -- gone. The re-execution cost plus downstream refresh failures pile up fast.

### 4. Timeout-Induced Full Reprocessing

OAuth tokens expire after ~1 hour. Scheduled refreshes timeout after 2 hours (5 hours in Premium). For a 500M-row semantic model refresh:

- Attempt 1: Runs 1h50m, times out. **All CUs wasted.**
- Attempt 2: Starts over, times out again. **All CUs wasted.**
- Engineer intervenes, enables incremental refresh. Succeeds in 20 minutes.

Nearly 4 hours of full-capacity CUs with zero value delivered. With duration trending and proactive alerts, this is caught weeks before the first timeout.

### 5. Duplicate Scheduled Runs

Common and invisible:
- A pipeline triggers a semantic model refresh AND the model has its own schedule. Double refresh.
- A notebook has both a workspace schedule and a pipeline trigger. Double execution.
- A Dataflow Gen2 is triggered by a pipeline AND has its own schedule.

Each duplicate consumes CUs for identical results. The Monitoring Hub shows each as a separate *successful* execution -- because it is. It just shouldn't have happened.

## Calculate Your Workspace's CU Waste

### Step 1: Collect Failure Data

Pull job instances for every item via the Fabric REST API:

```http
GET https://api.fabric.microsoft.com/v1/workspaces/{workspaceId}/items/{itemId}/jobs/instances
```

Calculate failure rate per item over the last 30 days. Any item above 5% is a significant waste source.

### Step 2: Find Retry Patterns

Look for job instances that start within minutes of a failed instance for the same item. Each retry is avoidable CU cost.

### Step 3: Find Duplicate Schedules

Query scheduled runs for each item and look for overlapping execution windows.

### Step 4: Track Duration Regression

Compare each item's P95 execution duration this week versus last month. Items up 20%+ are approaching timeout risk.

### Step 5: Define Your Waste Score

```
Waste Score = (Failed CU-seconds + Duplicate CU-seconds) / Total CU-seconds * 100
```

| Score | Assessment |
|-------|-----------|
| < 5% | Healthy |
| 5-10% | Investigate |
| 10-20% | Significant waste |
| > 20% | Likely experiencing regular throttling |

## Five Things You Can Do Today

**1. Create a duration baseline notebook.** Run weekly, query the Jobs API for every item, write P50/P95/max to a Lakehouse table. Alert when P95 increases > 15%.

**2. Audit schedule overlaps.** Build a time-grid of all pipeline triggers and notebook schedules. Find items executing more than once from different trigger sources.

**3. Enable incremental refresh for large datasets.** If any semantic model refresh takes > 45 minutes, evaluate incremental refresh. CU savings are typically 80-95% for large fact tables.

**4. Add failure alerting.** Use Power Automate to poll the Jobs API every 15 minutes and send a Teams notification on failure. Faster detection = fewer cascading downstream failures.

**5. Track your Waste Score weekly.** Make it a team metric. Review it in your weekly standup.

## Why This Matters More Than Money

The CU waste problem is about **capacity reliability**, not just dollars.

Every wasted CU-second reduces headroom for legitimate workloads. When headroom hits zero, throttling begins. Throttling creates failures. Failures create retries. Retries consume more CUs. This is a negative feedback loop that only breaks by either buying a bigger SKU or fixing root causes.

The teams that win in Fabric aren't the ones with the biggest SKUs. They're the ones that understand their consumption patterns, catch failures early, eliminate duplicate work, and trend duration metrics before they become timeouts.
